## Background

During Phase 2 execution (specifically plan 02-05 mid-execution at commit `3d85520`), the mono-crate approach for `kay-core` hit a structural wall at **1323 residual compile errors** that mechanical path-rewriting cannot resolve.

**Root cause:** ForgeCode's imported source was **23 separate crates**. D-01's original Option (a) "rename `lib.rs` → `mod.rs` and rewrite use paths" and Option (b) "thin shim" both fail for the same reasons:

| Error class | Count (post-02-05 partial) | Why mechanical fix fails |
|---|---|---|
| Expression-position path references (`forge_X::Y::z()` not in `use`) | ~600 E0433 | Rewriter only handles `use` statements |
| Missing dependencies in `kay-core/Cargo.toml` | ~200 E0432 (anyhow, thiserror, etc.) | Mono-crate needs union of all 23 crates' deps |
| Missing `include_str!` resource files | ~50 | Files exist in ForgeCode repo at paths relative to each original crate root, not present in `crates/kay-core/src/` |
| Proc macro self-reference (`forge_tool_macros`) | 2 | Rust forbids using a proc macro from the same crate that defines it |
| Trait object-safety violations (E0038) | 6 | Visibility semantics change under mono-crate consolidation |
| Associated-item bounds / ambiguous paths (E0220/E0223/E0107) | 43 | Cross-module naming collisions from flattening |

The `forgecode-parity-baseline` tag at commit `8af1f2b` was captured with the expectation that all 23 subtrees would eventually compile together. That was never true — ForgeCode's source only ever compiled as separate crates.

## Decision

**Execute D-01 Option (c)** from `.planning/phases/01-fork-governance-infrastructure/01-03-SUMMARY.md §Resolution`: split `kay-core` into multiple workspace sub-crates preserving ForgeCode's original 23-crate structure.

## Scope

### In scope

1. **Create 23 sub-crates** under `crates/kay-forge-*/` (or similar naming — planner decides), each containing one original `forge_*` subtree with its original `lib.rs` restored as the crate root.
2. **Revert plans 02-02..02-05 rewrites** — each sub-crate's internal `use crate::X` statements revert to the original forms because `crate` now refers to that sub-crate itself, not the mono `kay-core`.
3. **Dependency declarations per sub-crate** — each `crates/kay-forge-*/Cargo.toml` declares its own dependencies (anyhow, thiserror, tokio, etc.) instead of sharing kay-core's.
4. **Promote `forge_tool_macros` to `[lib] proc-macro = true`** in its own sub-crate; other sub-crates that consume it add it as a path dependency.
5. **Resource file placement** — `templates/`, `shell-plugin/`, `commands/`, `vertex.json`, `.forge.toml` fixtures move into the sub-crate that uses them (so `include_str!("../templates/...")` resolves relative to each sub-crate root).
6. **Workspace member list update** — `Cargo.toml` workspace `members` gains the 23 new entries and retires `kay-core` (or keeps it as an empty aggregator that re-exports from sub-crates).
7. **`kay-provider-openrouter` dependency update** — replace `kay-core = { workspace = true }` with specific sub-crate dependencies (likely `kay-forge-domain`, `kay-forge-services`, `kay-forge-repo`, `kay-forge-json-repair`).
8. **Remove `--exclude kay-core` from CI/docs/STATE** — this becomes reachable once all sub-crates compile cleanly (originally planned for 02-05 Task 2; absorbed here).
9. **Update `NOTICE`, `ATTRIBUTIONS.md`, `.forgecode-upstream-sha`** to reflect the sub-crate structure (the attribution SHA is unchanged — provenance integrity preserved).
10. **Restore Wave 0 test scaffold compile** — `kay-provider-openrouter/tests/common/mock_server.rs` + 6 SSE cassettes from plan 02-01 must continue to compile after the split.

### Out of scope

- `kay-provider-openrouter` implementation — remains Phase 2 plans 02-06..02-10 (unblocked after 2.5 lands)
- New `forge_*` functionality — sub-crate split is pure structural reorganization
- Re-tagging `forgecode-parity-baseline` — the existing tag at commit `8af1f2b` remains valid as the unmodified-import snapshot; Phase 11 re-tag (signed) still planned per D-OP-04
