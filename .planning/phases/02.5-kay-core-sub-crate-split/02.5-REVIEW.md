### Aggregator re-export pattern — security/correctness assessment

The intended `crates/kay-core/src/lib.rs` replaces 23 `pub mod forge_*;` declarations (pointing to now-moved directories, which would fail with E0583) with 6 `pub extern crate forge_*;` re-exports targeting `forge_api`, `forge_domain`, `forge_services`, `forge_repo`, `forge_config`, `forge_json_repair`.

- **`pub extern crate` is a safe, compile-time construct.** No `unsafe`, no FFI, no dynamic loading, no runtime side effects. It resolves purely through Cargo's dependency graph and the linker.
- **All 6 re-export targets exist as workspace members.** Verified against root `Cargo.toml` members list (lines 17, 26, 27, 30, 33, 35). Cargo will reject any `pub extern crate` whose crate name is not in the compilation unit's `[dependencies]`, so typo / shadowing attacks via `pub extern crate` are not possible once the sibling `kay-core/Cargo.toml` declares the path-deps.
- **No circular dependency risk.** kay-core → (forge_api, forge_domain, forge_services, forge_repo, forge_config, forge_json_repair). None of the 6 targets depend on kay-core (grep confirms kay-core is not declared as a path-dep in any `forge_*/Cargo.toml`). DAG preserved.
- **No information-disclosure change.** Per the threat model in `02.5-04-PLAN.md §T-2.5-07`, re-exporting forge_* internals is intentional (aggregator is for backward-compat consumers). forge_* crates are not published to crates.io.
- **Backward-compat surface is tight.** The aggregator re-exports only the 6 crates actually consumed downstream in Phase 2. The 17 other forge_* crates (forge_app, forge_walker, etc.) are intentionally NOT re-exported — a consumer that previously used `kay_core::forge_walker::...` must migrate. This is a deliberate API surface reduction, not a silent breakage, and is documented in the aggregator's doc-comment.

### kay-core/Cargo.toml path-deps

The intended manifest declares:

```
forge_api         = { path = "../forge_api" }
forge_domain      = { path = "../forge_domain" }
forge_services    = { path = "../forge_services" }
forge_repo        = { path = "../forge_repo" }
forge_config      = { path = "../forge_config" }
forge_json_repair = { path = "../forge_json_repair" }
```

- **Relative paths only.** No absolute paths that could leak dev-machine layout.
- **No `version =`, `registry =`, or `git =` fallback.** Pure `path =` means Cargo will error if the sibling crate is missing — no silent fallback to a registry typosquat. Workspace member enforcement is the primary defense; path-only dep is the secondary.
- **No `features = [...]` overrides.** Downstream consumers of kay-core get the default feature set of each sub-crate, which matches what ForgeCode itself expects.
- **No version drift risk.** All 6 targets inherit `version.workspace = true`, so pinning is single-source at workspace root.

### kay-provider-openrouter/Cargo.toml path-deps

The intended `[dependencies]` block declares 5 forge_* path-deps plus 8 workspace crates:

```
forge_domain, forge_config, forge_services, forge_repo, forge_json_repair
tokio, reqwest, reqwest-eventsource, serde, serde_json, anyhow, thiserror, tracing
```

- **No forge_app or forge_api dep.** Good — the provider HAL should consume the narrow domain/services surface, not the top-level API crate (which would pull forge_infra and the whole service graph transitively and couple the HAL to the integration shell).
- **Every non-forge dep uses `{ workspace = true }`.** Zero inline version pins in the HAL → no drift between HAL and the harness.
- **`[dev-dependencies]` untouched** (mockito, proptest, pretty_assertions, serde_json, tokio test-util). The Wave 0 test scaffold from plan 02-01 continues to compile unchanged per plan 02.5-04 acceptance criteria.
- **No transitive security escalation.** The provider HAL gains access to `reqwest`, `reqwest-eventsource`, `tokio` — all already in the workspace root pinning — at the same rustls-only, hickory-dns, http2 feature surface that ForgeCode upstream uses. No `native-tls` fallback (protection against mixed TLS stacks).

## Info

### IN-01: `forge_main` inline `update-informer` pin shadows workspace pin

**File:** `crates/forge_main/Cargo.toml:54`

**Issue:** `forge_main` declares `update-informer = { version = "1.2.0", default-features = false, features = ["github", "ureq", "rustls-tls"] }` as an inline version pin, shadowing the workspace pin at root `Cargo.toml:154` (`update-informer = "1.2.0"`). The version literal is identical today, but a future workspace bump would not flow into forge_main until someone notices the drift. Every other shared dep in this manifest uses `{ workspace = true }`.

**Fix:** Promote the feature set to the workspace pin and consume via workspace inheritance:

```toml
# In root Cargo.toml [workspace.dependencies]
update-informer = { version = "1.2.0", default-features = false, features = ["github", "ureq", "rustls-tls"] }

# In crates/forge_main/Cargo.toml
update-informer = { workspace = true }
```

Not a correctness issue; flagged only because the rest of the manifest uses inheritance consistently. Hygiene-only.

### IN-02: On-disk state vs. target state

**File:** `crates/kay-core/src/lib.rs` (working tree) and `crates/kay-core/Cargo.toml` (working tree)

**Issue:** As of this review, the on-disk `kay-core/src/lib.rs` still contains the 23 `pub mod forge_*;` declarations (lines 20-42) pointing to directories that no longer exist (they were moved in Waves 0-6). The on-disk `kay-core/Cargo.toml` still has an empty `[dependencies]` block. Plan 04 Task 2 will rewrite both atomically. Until that commit lands, `cargo check -p kay-core` would fail with 23 × E0583 errors — which is why the CI `--exclude kay-core` flag (scheduled for removal in Plan 04 Task 3) is still in place.

**Fix:** Ensure the Plan 04 Task 2 commit stages **both** the `lib.rs` rewrite AND the `Cargo.toml` rewrite atomically. Do not commit the `pub extern crate` rewrite without the matching `path =` declarations, or the crate will fail to compile with `E0432: unresolved import`. The plan already specifies a single combined commit; this is a note, not a bug.
