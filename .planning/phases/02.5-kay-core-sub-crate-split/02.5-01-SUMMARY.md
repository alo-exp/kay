## Decisions Made

1. **Fetch shell-plugin/lib/actions/ even though plan didn't list it.** `forge_main/zsh/plugin.rs:15` uses `include_dir!("$CARGO_MANIFEST_DIR/../../shell-plugin/lib")` which recursively embeds every file in the directory tree. Skipping the `actions/` subdirectory would have left the embedded FS incomplete and broken the ZSH plugin bootstrap at runtime. Applied Rule 2 (missing critical functionality).

2. **Skip `git restore .` from Task 1 STEP 1.** Plan anticipated 132 dirty files under `crates/kay-core/src/` but at resume time only `.planning/STATE.md` and `.planning/ROADMAP.md` were dirty — both orchestrator-owned. Blanket `git restore .` would have destroyed legitimate orchestrator state. Per resume-state instruction, skipped the step and proceeded directly to verifying the already-staged Task 1 revert was correct.

3. **Use Python to parse GitHub contents API JSON.** The plan's shell one-liner (`grep '"download_url"' | sed ...`) processed `"download_url": null` entries (directories) as if they were files, producing a spurious output path. Switched to `python3` with explicit `type == 'file'` filtering for the `actions/` fetch. The initial `shell-plugin/lib/` fetch had already completed with only a harmless curl error (malformed URL rejected), so no cleanup was needed — no bogus file was written to disk.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing Critical Functionality] Fetch shell-plugin/lib/actions/ subdirectory**
- **Found during:** Task 2 (STEP 3 directory fetch)
- **Issue:** Plan's fetch script only listed files directly in `shell-plugin/lib/`. But `forge_main/zsh/plugin.rs:15` uses `include_dir!` which recursively embeds all subdirectories — so `actions/` contents are required at compile time. Without them, the compiled binary's embedded filesystem would be missing 7 action handlers (auth, config, conversation, core, editor, git, provider) that the dispatcher needs to function.
- **Fix:** Added a second GitHub contents API fetch for `shell-plugin/lib/actions?ref=022ecd9`, downloaded all 7 `.zsh` files into `shell-plugin/lib/actions/`.
- **Files modified:** Added 7 files under `shell-plugin/lib/actions/` (commit e299e79)
- **Verification:** `find shell-plugin -type f | wc -l` = 18 (4 root + 7 lib + 7 actions). `include_dir!` expansion at compile time will now find the full tree.
- **Committed in:** e299e79 (Task 2 commit — included in same chore commit as the rest of Task 2)

**2. [Rule 3 — Blocking Issue] Skip `git restore .` in Task 1 STEP 1**
- **Found during:** Task 1 STEP 1 (by prior executor, resume session confirmed)
- **Issue:** Plan's `git restore .` would have reset `.planning/STATE.md` and `.planning/ROADMAP.md`, which are orchestrator-owned and carry current Phase 2.5 state not yet committed. Blanket restore would have destroyed orchestrator tracking.
- **Fix:** Skipped `git restore .` entirely. The plan's original motivation (discard 132 stale kay-core working-tree files from `/tmp/rewrite-indented-02-05.sh`) no longer applied — those files had already been cleared before Task 1 began.
- **Files modified:** None (this was an omission, not an addition)
- **Verification:** `git diff --cached --stat` after staging the revert showed exactly the expected 295 files, none of them planning files. Planning files remained unstaged (correct).
- **Committed in:** N/A (no commit needed for this deviation; it's a skipped step)

**3. [Rule 1 — Bug] Use python3 for GitHub API JSON parsing**
- **Found during:** Task 2 STEP 3 (shell-plugin/lib/ directory fetch)
- **Issue:** Plan's shell pipeline (`grep '"download_url"' | sed ...`) matched a `"download_url": null` entry for the `actions` subdirectory and passed "null" to `basename`, which produced a malformed URL. curl rejected it with exit 3 "URL rejected: Malformed input to a URL function" — no bogus file was written, but the error was disruptive.
- **Fix:** For the subsequent `actions/` fetch, replaced the grep|sed pipeline with `python3` parsing that filters by `type == 'file'` before extracting `download_url`. This robustly handles directories.
- **Files modified:** None (approach change only)
- **Verification:** Actions fetch completed cleanly with 7 files, no curl errors.
- **Committed in:** N/A (technique-only change, not a code change)
