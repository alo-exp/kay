# Phase 2.5: kay-core sub-crate split — Research

**Researched:** 2026-04-20
**Domain:** Rust workspace multi-crate restructuring / proc-macro isolation / include_str! path resolution / git revert sequencing
**Confidence:** HIGH (all structural claims VERIFIED from live codebase and upstream ForgeCode GitHub at the canonical SHA; no significant ASSUMED claims)

---

## Summary

Phase 2.5 is a pure structural surgery: disassemble the `kay-core` mono-crate back into the 23 workspace sub-crates that ForgeCode's source was designed for. The root cause of 1323 residual compile errors is not bad path-rewrite tooling — it is that ForgeCode's source was written as 23 cooperating crates and cannot compile correctly as a single monolith. The proc-macro self-reference error is the clearest evidence: Rust's compiler enforces that a proc-macro crate cannot be used in the same compilation unit that defines it.

The research confirms all six error classes from the CONTEXT.md are real and cannot be overcome mechanically within the mono-crate shape. The path forward is verified: create 23 workspace member crates under `crates/<name>/`, each containing exactly the subtree that was imported in Phase 1, with the `lib.rs` restored as the crate root. Reverse the path-rewrite commits from plans 02-02 through 02-05 (since each sub-crate's `use crate::X` is now correct relative to its own crate root). Redistribute dependencies from the current empty `kay-core/Cargo.toml` to each sub-crate's Cargo.toml using the upstream ForgeCode Cargo.toml files as canonical source.

The critical finding that the planner must budget for: **four resource directories (`templates/`, `shell-plugin/`, `commands/`, and root-level `vertex.json`) exist in the ForgeCode repo root but were never imported into Kay's repo.** The `include_str!` calls in `forge_main`, `forge_services`, `forge_domain`, and `forge_repo` reference them. Phase 2.5 must either fetch and place these files or rewrite those `include_str!` calls. Similarly, a `.forge.toml` file that `forge_config/reader.rs` expects at the crate root is absent.

**Primary recommendation:** Use naming scheme `crates/forge_<name>/` (package name `forge_<name>`) for all 23 sub-crates, exactly matching ForgeCode's original layout. Do not add a `kay-` prefix to the forge_* sub-crates — these are internal implementation crates, not public-facing API. Keep the Kay-branded crates (`kay-cli`, `kay-provider-openrouter`, etc.) with the `kay-` prefix. This avoids double-renaming and makes the parity baseline traceable.

### Path resolution math for missing files (when sub-crate is at `crates/<name>/`):

When `forge_main` lives at `crates/forge_main/`, `CARGO_MANIFEST_DIR` = `crates/forge_main/`.
Source file `src/zsh/plugin.rs` uses relative `include_str!` paths:
- `"../../../../shell-plugin/..."` from `src/zsh/` = 4 levels up from the file's directory:
  - 1: `src/zsh/` → `src/`
  - 2: `src/` → `crates/forge_main/`
  - 3: `crates/forge_main/` → `crates/`
  - 4: `crates/` → repo root
  - Result: `<repo_root>/shell-plugin/` — matches ForgeCode original.

- `include_dir!("$CARGO_MANIFEST_DIR/../../shell-plugin/lib")` from `crates/forge_main/`:
  - `crates/forge_main/../../shell-plugin/lib` = `shell-plugin/lib` at repo root — matches.

When `forge_repo` lives at `crates/forge_repo/`:
- `concat!(env!("CARGO_MANIFEST_DIR"), "/../../vertex.json")` = `crates/forge_repo/../../vertex.json` = repo root `vertex.json` — matches.

When `forge_services` lives at `crates/forge_services/`:
- `"../../../commands/..."` from `src/command.rs`:
  - 1: `src/` → `crates/forge_services/`
  - 2: `crates/forge_services/` → `crates/`
  - 3: `crates/` → repo root
  - Result: `<repo_root>/commands/` — matches ForgeCode original.

When `forge_domain` lives at `crates/forge_domain/`:
- `"../../../../templates/..."` from `src/tools/result.rs`:
  - 1: `src/tools/` → `src/`
  - 2: `src/` → `crates/forge_domain/`
  - 3: `crates/forge_domain/` → `crates/`
  - 4: `crates/` → repo root
  - Result: `<repo_root>/templates/` — matches ForgeCode original.

When `forge_config` lives at `crates/forge_config/`:
- `"../.forge.toml"` from `src/reader.rs`:
  - 1: `src/` → `crates/forge_config/`
  - Result: `crates/forge_config/.forge.toml` — this is the crate root .forge.toml.

**Conclusion:** All `include_str!` paths resolve correctly once sub-crates are at `crates/<name>/` — exactly mirroring ForgeCode's original structure. ## 8. Naming Convention Recommendation

### Options analyzed

**Option A: `crates/forge_<name>/` with package name `forge_<name>`**
- Matches ForgeCode's original layout exactly
- `include_str!` relative paths resolve identically to ForgeCode upstream
- No double-renaming (source is already `forge_*` named)
- Traceability: `git log -- crates/forge_domain/` maps cleanly to ForgeCode's `crates/forge_domain/`
- The internal/implementation nature is clear; users import `forge_domain::X`, not `kay_forge_domain::X`

**Option B: `crates/kay-forge-<name>/` with package name `kay-forge-<name>`**
- Adds Kay branding to internal crates
- Avoids name collision with ForgeCode if both are ever in the same workspace (not a concern for Kay)
- Requires renaming directories, adding one more transformation layer
- Makes imports more verbose: `use kay_forge_domain::X` instead of `use forge_domain::X`
- Does NOT simplify `include_str!` path math (the path resolution depends on depth, not name)

**Recommendation: Option A — `crates/forge_<name>/` with package name `forge_<name>`**

Rationale:
1. ForgeCode's `include_str!` paths resolve correctly when the crate is at `crates/<name>/` — verified by path math above.
2. The Phase 2 plans 02-06..02-10 already reference types as `forge_domain::X`, `forge_services::Y` in their import examples. Option A requires no adjustments to those plans.
3. The CONTEXT.md states plans 02-06..02-10 "reference specific forge_* modules via typed paths" and will need only "minor import path adjustments where `kay_core::forge_X::Y` becomes `kay_forge_X::Y`." Option A makes this adjustment even simpler: `kay_core::forge_X::Y` → `forge_X::Y`.
4. Kay's public-facing crates (`kay-cli`, `kay-provider-openrouter`, `kay-core` aggregator) retain the `kay-` prefix. The `forge_*` crates are implementation details, not public API.

**Exception:** `forge_main` is special — it becomes the CLI binary crate. At Phase 5, it will be rebranded to `kay-cli` (CONTEXT.md CLI-04: "rebrands `forge_main`"). For now in Phase 2.5, it stays `forge_main` at `crates/forge_main/` — the rebrand is Phase 5 scope.

## 10. Risk and Pitfall Analysis

### Pitfall 1: Topological ordering violation
**What goes wrong:** Adding a sub-crate to the workspace before its forge_* dependencies are in the workspace causes `cargo metadata` to fail with "package not found."
**Root cause:** Cargo's workspace resolver requires all path-dependency targets to be registered workspace members.
**Prevention:** Strictly follow the Wave order in §1. Add Wave 0 crates first, verify `cargo check` on each, then proceed to Wave 1.
**Warning signs:** `error[E0463]: can't find crate for 'forge_domain'` or workspace-level `package not found in workspace` errors.

### Pitfall 2: `forge_tool_macros` missing `proc-macro = true`
**What goes wrong:** The proc-macro crate compiles as a normal library; `#[proc_macro_derive(ToolDescription)]` silently does nothing and `forge_domain` get mysterious "attribute macro not found" or "function-like proc macro not found" errors.
**Root cause:** Without `[lib] proc-macro = true`, Cargo does not link the crate as a proc-macro.
**Prevention:** First thing in `crates/forge_tool_macros/Cargo.toml`: add `[lib]\nproc-macro = true`.
**Warning signs:** `error: cannot find derive macro 'ToolDescription' in this scope` in `forge_domain`.

### Pitfall 3: Missing resource files for `include_str!`
**What goes wrong:** `cargo check` on `forge_main`, `forge_services`, `forge_domain`, `forge_repo`, or `forge_config` fails with `error: couldn't read <file>: No such file or directory` at compile time.
**Root cause:** Phase 1 only imported `crates/forge_*/src/` content. The resource files at ForgeCode's repo root (`shell-plugin/`, `templates/`, `commands/`, `vertex.json`) and at `crates/forge_config/` (`.forge.toml`) were not copied.
**Prevention:** Plan 02.5-01 must fetch these from upstream SHA before creating any sub-crate. Verify each file exists at the path the `include_str!` resolves to.
**Warning signs:** `error: couldn't read templates/forge-partial-tool-error-reflection.md` or similar.

### Pitfall 4: `rand` version mismatch
**What goes wrong:** `forge_spinner` needs `rand 0.10.0`; Kay's Cargo.lock has `rand 0.9.4`. If Kay's workspace pins `rand` at `"0.9"`, `forge_spinner` fails to compile because `rand 0.10` has breaking API changes.
**Root cause:** Version incompatibility between what was in Kay's workspace (rand 0.9.x, pulled in by proptest) and ForgeCode's requirement.
**Prevention:** Add `rand = "0.10.0"` to `[workspace.dependencies]`. Proptest can work with rand 0.9 in its own dep tree (Cargo resolves them as separate nodes if the SemVer ranges don't overlap — but 0.9 vs 0.10 don't overlap so both versions will coexist, which is fine for dev-deps).
**Warning signs:** `error[E0277]: the trait bound 'StdRng: rand_core::RngCore' is not satisfied` or similar API surface errors.

### Pitfall 5: `diesel` requires native SQLite library linkage
**What goes wrong:** `forge_infra` and `forge_repo` depend on `diesel` with `sqlite` feature. On CI Linux and Windows, `libsqlite3` may not be installed. `libsqlite3-sys 0.36.0` with `bundled` feature compiles SQLite from source — but requires a C compiler.
**Root cause:** `diesel` with `features = ["sqlite"]` dynamically links to libsqlite3 by default; `libsqlite3-sys` with `bundled` statically compiles it.
**Prevention:** Use `libsqlite3-sys` with `features = ["bundled"]` in Kay's workspace.dependencies, exactly as ForgeCode specifies. CI must have a C/C++ toolchain (present on ubuntu-latest, macos-latest, windows-latest by default).
**Warning signs:** `error: failed to run custom build command for 'libsqlite3-sys'`.

### Pitfall 6: `forge_repo` has a build.rs / tonic gRPC codegen
**What goes wrong:** `forge_repo`'s `tonic-prost-build` build dependency generates Rust code from `.proto` files at build time. If the `.proto` files are not present (they may be in `crates/forge_repo/src/` as non-Rust files), the build fails.
**Root cause:** tonic-prost-build runs in `build.rs` and requires `.proto` source files at known paths.
**Prevention:** When moving `forge_repo` content from `crates/kay-core/src/forge_repo/` to `crates/forge_repo/src/`, verify all non-`.rs` files (including any `*.proto`, `build.rs`) are also moved.
**Warning signs:** `error: .proto file not found` during `cargo check -p forge_repo`.

### Pitfall 7: `kay-core/src/lib.rs` becomes obsolete
**What goes wrong:** After the split, `crates/kay-core/src/lib.rs` still has `pub mod forge_api;` etc., which will look for `crates/kay-core/src/forge_api/` — but those directories have been moved out. This causes E0583 errors on `kay-core` itself.
**Root cause:** The workspace members list still includes `kay-core`, and its lib.rs references paths that no longer exist.
**Options:**
- (a) Remove `kay-core` from workspace members entirely and delete the crate.
- (b) Replace `kay-core/src/lib.rs` with a minimal re-exporter that does `pub use forge_domain; pub use forge_services; ...` — useful for backward compat if Phase 2 plans reference `kay_core::*`.
- (c) Delete `kay-core` crate directory.

**Recommendation:** Option (b) — keep `kay-core` as an empty aggregator with `pub extern crate` re-exports. This avoids breaking `kay-provider-openrouter` and any future code that may reference `kay_core::forge_X`. Phase 5 can retire it cleanly when `forge_main` → `kay-cli` rebrand lands.
**Warning signs:** `error[E0583]: file not found for module 'forge_api'` on `cargo check -p kay-core`.

### Pitfall 8: Wave 0 test scaffold breakage
**What goes wrong:** `cargo check -p kay-provider-openrouter --tests` fails because the test imports or dev-deps break after the restructuring.
**Root cause:** plan 02-01's MockServer is deliberately self-contained (no `kay-core` import), so this risk is LOW. However, `Cargo.lock` churn from adding 23 new workspace members may cause unexpected re-resolution of dev-dep versions.
**Prevention:** Run `cargo check -p kay-provider-openrouter --tests` as a gate after EVERY plan in Phase 2.5, not only at the end. This gives early signal.
**Warning signs:** Compilation errors in `crates/kay-provider-openrouter/tests/`.

### Pitfall 9: `include_dir!` macro in `forge_main/zsh/plugin.rs`
**What goes wrong:** `include_dir!("$CARGO_MANIFEST_DIR/../../shell-plugin/lib")` uses the `include_dir` crate (not the std `include_str!`). This macro embeds an entire directory at compile time. If `shell-plugin/lib/` doesn't exist or is empty, the macro silently embeds nothing, which may cause a runtime failure.
**Root cause:** `forge_main` uses both `include_str!` and `include_dir!` for shell-plugin content.
**Prevention:** Confirm `shell-plugin/lib/` exists with `.zsh` files when fetching from upstream. The `forge_embed` crate provides the `forge_embed::files()` helper that iterates an embedded `Dir`.
**Warning signs:** Runtime panic when calling `generate_zsh_plugin()`.

### Pitfall 10: Cross-crate visibility changes
**What goes wrong:** Types or functions that were `pub(crate)` in the mono-crate become truly private within their new sub-crate (correct) but some inter-crate callers relied on them being accessible. Or vice-versa: items that should have been `pub` within the mono-crate context now need to be `pub` in the sub-crate to be importable by dependents.
**Root cause:** `pub(crate)` in sub-crate X is not visible to sub-crate Y, even if Y depends on X. In the mono-crate, `pub(crate)` was visible across all 23 subtrees.
**Prevention:** When `cargo check -p forge_Y` fails with "function is private" or "module is private" pointing into `forge_X`, promote the item from `pub(crate)` to `pub` in `forge_X`. Document each change.
**Warning signs:** `error[E0603]: function 'xyz' is private` or `error[E0616]: field 'xyz' of struct 'Foo' is private`.

### Rust proc-macro isolation rule [VERIFIED: Rust Reference]

Rust enforces at the compiler level: a crate with `[lib] proc-macro = true` can only be used as a proc-macro (not as a regular library in the same crate). Any crate that defines a `#[proc_macro_derive]` must be a separate compilation unit from any crate that invokes that derive. This is why `forge_tool_macros` must be the first forge_* sub-crate added to the workspace.
