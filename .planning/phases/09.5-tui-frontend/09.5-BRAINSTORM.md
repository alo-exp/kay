# Brainstorm: Phase 9.5 — TUI Frontend (ratatui)

## Problem Statement

Kay needs a full-screen terminal UI (TUI) that consumes the same `kay-cli` JSONL event stream as the Tauri GUI. Users running over SSH, in low-bandwidth terminals, or by preference get a keyboard-driven, ANSI-256-color experience with live session view, tool-call inspector, and cost meter. No parallel agent-loop implementation — TUI is a pure frontend over the CLI contract.

## Approaches Considered

### Approach A: Ratatui + kay-cli child-process (subprocess approach)

- `kay-tui` spawns `kay run --jsonl` as a subprocess
- Reads `stdout` (JSONL events) via async pipe
- ratatui renders the live transcript
- TUI sends signals (stop/pause) via stdin or process signals

**Pros:**
- Cleanest separation — TUI is purely a UI layer
- Zero coupling to agent loop internals
- Can work with any `kay` binary that emits JSONL
- Easy to test: just feed JSONL fixtures to ratatui
- ratatui is actively maintained (ratatui 0.27+)

**Cons:**
- IPC latency: subprocess boundary vs. in-process channel
- Requires `--jsonl` flag on `kay run` (already planned per CLI-05)
- Stop signal requires separate mechanism (signal vs. stdin)
- Subprocess lifecycle management (restart on crash)

**Complexity:** 2/5 — Ratatui widgets + child process + async read

---

### Approach B: Ratatui + shared library (embedded approach)

- `kay-tui` links against `kay-core` directly
- Creates agent loop in-process, renders events via shared memory or channel
- ratatui renders the live transcript
- No subprocess — everything in one process

**Pros:**
- Zero IPC latency
- TUI can control agent loop directly
- Shares context engine, session store, etc.
- Can expose more internals (e.g. sandbox status, verification state)

**Cons:**
- Heavier coupling: `kay-tui` becomes a full app, not a pure UI
- Build complexity: ratatui requires `crossterm` + `tauri` features don't mix cleanly
- Harder to test independently
- Duplicates agent-loop lifecycle management in TUI

**Complexity:** 4/5 — requires embedding agent loop + ratatui in same process

---

### Approach C: Kay-tui as thin ratatui wrapper over kay-cli (pipeline approach)

- `kay-tui` pipes `kay run --jsonl` output like Approach A
- But uses `kay-cli`'s existing `--stream-format=jsonl` output
- ratatui renders live transcript
- Shared `kay-core` types for deserialization (avoid re-defining IpcAgentEvent in TUI)

**Pros:**
- Cleanest separation (like A)
- Reuses existing CLI-05 contract (like B)
- ratatui + kay-cli pipeline is the same pattern as Tauri GUI (they both front-end the CLI contract)
- ratatui 0.27+ is stable and widely used

**Cons:**
- Same subprocess latency as A
- Ratatui async input handling requires careful state management

**Complexity:** 2/5 — same as A but leverages existing CLI contract

---

## Recommendation

**Approach A (Ratatui + kay-cli subprocess)** — same pipeline as the Tauri GUI:
- Tauri GUI uses `ipc::Channel<IpcAgentEvent>` (in-process IPC)
- TUI uses `kay run --jsonl` subprocess (pipe IPC)
- Both consume the same `kay-cli` JSONL event contract
- ratatui 0.27+ with `crossterm` backend
- `tokio::process::Command` to spawn `kay run --jsonl`
- ratatui `AsyncReader` or `mpsc` channel to feed JSONL events to render loop

**Rationale:** The Tauri GUI and TUI are both frontends consuming the same CLI contract. Approach A is the most consistent with the Phase 9 design (no `externalBin` sidecar; single merged binary; CLI is the contract). Approach B couples TUI too tightly to internals and requires significant build complexity.

**Key decisions:**
- ratatui 0.27+ with `crossterm` backend
- Subprocess: `tokio::process::Command` spawning `kay run --jsonl --prompt "..." --persona ...`
- Event feed: `tokio::io::AsyncReadExt::read_to_string` on subprocess stdout → JSONL parser
- TUI state: `App` struct with `ratatui::Frame`, `Vec<IpcAgentEvent>`, `SessionMetrics`
- Stop/resume via `CancellationToken` → SIGTERM to subprocess
- Key bindings: standard ratatui navigation (arrows, Enter, Esc, q/quit)
