## Dimensions

### 1. Modularity ✅ PASS

| Check | Status | Notes |
|-------|--------|-------|
| Each file has one clear purpose | ✅ | Spec §5 defines clean module boundaries: `app.rs`, `events.rs`, `jsonl.rs`, `subprocess.rs`, `components/*` |
| Files >300 lines → split | ✅ | No file in spec exceeds 300 lines |
| Public API minimal | ✅ | Only `TuiEvent`, `TuiToolOutputChunk`, `TuiVerificationOutcome` are `pub` |
| No circular deps | ✅ | `kay-tui` → `kay-core`; no reverse deps |

**Finding:** MOD-01: `TuiEvent` is a re-definition of `IpcAgentEvent` from `kay-tauri`. The wire format is shared but the type lives in two crate boundaries. This is intentional (same rationale as `IpcAgentEvent` mirroring `AgentEvent`) but introduces a maintenance risk: future changes to the wire format must be mirrored in both `kay-tauri/src/ipc_event.rs` and `kay-tui/src/events.rs`.

**Mitigation (recorded):** Add a comment in both files referencing the canonical other file. Consider a shared `kay-types` crate in Phase 12 to consolidate the wire type definitions.

### 2. Error Handling ⚠️ PARTIAL (pre-plan — runtime evaluation)

| Check | Status | Notes |
|-------|--------|-------|
| `Result<T, E>` for fallible ops | ✅ | Subprocess: `tokio::process::Command` + error propagation via `?` |
| Never `.unwrap()` in lib | ⚠️ | Spec §3.3 shows `std::thread::sleep` (not fallible); ratatui render errors will use `Result<_, ratatui::error::Error>` |
| Propagate errors with `?` | ✅ | Implied by subprocess pattern |

**Finding:** ERR-01: JSONL parsing with `serde_json` from untrusted subprocess output — if `kay run` emits malformed JSON (e.g. log lines mixed with JSONL), the parser must handle it gracefully. Spec does not address this.

**Mitigation:** JSONL parser should skip non-JSON lines silently (log lines to stderr, not stdout). Add `skip malformed` logic.

### 5. Security ⚠️ PARTIAL (pre-plan)

| Check | Status | Notes |
|-------|--------|-------|
| No API keys in code | ✅ | `kay-tui` spawns subprocess; keys via environment |
| No injection risks | ✅ | Subprocess args use `Command::new()` with no shell expansion |
| Ratatui input sanitization | ⚠️ | ratatui key events are `enum`s — no string injection surface |
| Subprocess lifecycle security | ⚠️ | SIGTERM to subprocess is correct; SIGKILL as fallback after 5s |

**Finding:** SEC-01: `tokio::process::Command` with user-provided prompt — if prompt contains shell metacharacters, `kay run --jsonl` receives them as-is (no shell expansion). However, `kay run` parses them as program arguments, not shell commands. No shell injection. ✅

**Finding:** SEC-02: The TUI passes `--prompt` to `kay run`. If the prompt contains newlines, `Command::arg()` passes them correctly — no shell interpolation. ✅

### 6. Performance ⚠️ PARTIAL

| Check | Status | Notes |
|-------|--------|-------|
| No unbounded memory growth | ⚠️ | Event log unbounded in spec — need `EventLog` with LRU eviction or ring buffer |
| JSONL parsing amortized O(n) | ✅ | Line-by-line parsing; `BufRead::read_line` is O(line_length) |
| ratatui render at 60fps | ⚠️ | `ratatui::draw` called on `Backend::render` loop; no animation — events are sparse |

**Finding:** PERF-01: Event log grows unbounded for long sessions. Must implement LRU cap (e.g. `EventLog::push` evicts oldest when len > 10,000 events). Recording as deferred implementation detail.
