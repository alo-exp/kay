ForgeCode has 23 workspace crates under `crates/forge_*` [VERIFIED per 01-RESEARCH.md].
Per D-05 ("kay-core = imported ForgeCode source" — singular), the Phase 1 decision is:

**PRESERVE ForgeCode's internal src/ structure but import the union into a SINGLE `crates/kay-core/src/` tree.** Concretely:
- Copy ForgeCode's `crates/forge_*/src/**/*.rs` → `crates/kay-core/src/` flattened per-crate under subdirectories named after each upstream crate. Example:
  - `forgecode/crates/forge_api/src/lib.rs` → `crates/kay-core/src/forge_api/lib.rs`
  - `forgecode/crates/forge_domain/src/model.rs` → `crates/kay-core/src/forge_domain/model.rs`
- Adjust kay-core/src/lib.rs (from plan 01) to declare each forge_* subtree as a module: `pub mod forge_api; pub mod forge_domain; …`
- Preserve each forge_* subdirectory's own `Cargo.toml` NOT imported at crate level — they become inert source files under kay-core for this phase. Kay's own Cargo.toml is the only active manifest for kay-core.
- Copy ForgeCode's workspace-root `.rustfmt.toml` fields verbatim — already done in plan 01.
- Do NOT copy ForgeCode's `.github/workflows/**` (Kay's CI is separate per 01-RESEARCH.md §ForgeCode-Inherited File Audit).
- Do NOT copy ForgeCode's `LICENSE`, `NOTICE` (absent upstream), `README.md`, `CONTRIBUTING.md`, `SECURITY.md` — Kay has its own.

This keeps a single `kay-core` crate (satisfying D-05's "singular" phrasing) while preserving ForgeCode's internal module names for blame-level traceability.
