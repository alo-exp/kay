# Design: Phase 9.1 — Comprehensive Test Coverage

**Date:** 2026-04-23  
**Phase:** 9.1  
**Requirements:** TEST-01..TEST-07  
**Status:** Rev 1 — initial draft

---

## 1. Overview

Kay's well-tested core crates (`kay-core`, `kay-tools`, `kay-cli`, `kay-verifier`, `kay-session`, `kay-context`, `kay-provider-openrouter`) have comprehensive integration test suites. Phase 9.1 closes the remaining test gaps: 18 `forge_*` crates with zero integration test directories, 3 `kay-sandbox-*` crates with only 1 unit test each, `kay-tauri` with only binding and memory canary tests, and `kay-tui` as a stub with no tests at all. The result is a workspace where every crate has at least one integration test in `tests/` and every pub API function called by Kay code is covered.

---

## 2. Goals

- Every crate in `crates/` has at least one integration test file in a `tests/` directory.
- All `forge_*` pub API functions called by Kay code have test coverage (integration-level, not just unit).
- `kay-sandbox-{linux,macos,windows}` have real subprocess escape tests on the respective OS, included in the CI matrix on macOS + Linux + Windows.
- `kay-tauri` IPC command handlers (`start_session`, `stop_session`, `get_session_status`) are tested via `tauri::test::mock_builder()`.
- `kay-tauri` UI has a `tauri-driver` + WebDriverIO smoke suite passing in CI.
- `kay-tui` has at minimum a compile test and a basic `ratatui::backend::TestBackend` render test.
- CI test matrix runs on `ubuntu-latest` + `macos-14` + `windows-latest`.
- Coverage gate: CI fails if any gap-list crate has zero tests post-merge.

---

## 3. Non-Goals

- **No source changes to `forge_*` crates** beyond adding `tests/` directories and test files. `forge_*` behavior is assumed correct; this phase only adds test coverage.
- **No new features** — no new public APIs, no new crates.
- **No performance optimization** — no benchmarks or profiling unless required by a failing test.
- **No `forge_*` refactoring** — tests are additive only.
- Well-tested crates (kay-core, kay-tools, kay-cli, kay-verifier, kay-session, kay-context, kay-provider-openrouter) are excluded from the gap list.

---

## 4. Personas & Value

### Contributor
Writes a feature that calls a function from a `forge_*` crate. After Phase 9.1, the contributor can run `cargo test -p <crate>` and immediately see whether their call site is covered by an integration test.

### CI Operator
Maintains the CI pipeline. After Phase 9.1, the CI matrix covers all 3 OSes for sandbox tests, and the coverage gate prevents regressions where a new crate lands without tests.

### Benchmark Runner
Runs `cargo test --workspace` before profiling runs. After Phase 9.1, all crates have tests that execute fast enough to include in the pre-profiling smoke run.

## 8. Requirements

### TEST-01: Every crate has >= 1 integration test in tests/ dir

**Statement:** Every crate listed in the gap list (Table in §6) has at least one integration test file in `crates/<crate>/tests/<name>.rs`.

**Acceptance criteria:**
- `tests/` directory created in each gap-list crate
- At least one `#[test]` function in at least one `.rs` file under `tests/`
- `cargo test -p <crate>` passes for each gap-list crate on the crate's native OS
- No new source code changes in `forge_*` crates beyond test files

---

### TEST-02: All forge_* pub API functions called by Kay have test coverage

**Statement:** Every `pub fn` / `pub async fn` / `pub struct` / `pub enum` in each `forge_*` crate that is called by any Kay crate (`kay-*`) has at least one integration test exercising that API surface.

**Acceptance criteria:**
- Audit pass: `grep -r "forge_" crates/ --include="*.rs" | grep -v tests/ | grep "use\|pub use"`, cross-referenced against `crates/forge_*/src/lib.rs` exports
- For each exported `pub fn` called by Kay code: at least one integration test calling that function
- Integration test files named by feature area (e.g., `tests/read.rs`, `tests/write.rs`)
- `cargo test -p forge_fs` passes (example for forge_fs)

---

### TEST-03: Sandbox crates have real subprocess escape tests on macOS + Linux + Windows

**Statement:** Each of `kay-sandbox-{linux,macos,windows}` has at least one integration test that spawns a real subprocess and verifies that sandbox enforcement (deny write to `/etc/passwd`, deny exec of `/bin/sh`, etc.) fires correctly. These tests run in CI on the respective OS.

**Acceptance criteria:**
- `kay-sandbox-linux`: tests run on `ubuntu-latest` CI
- `kay-sandbox-macos`: tests run on `macos-14` CI
- `kay-sandbox-windows`: tests run on `windows-latest` CI
- Each test spawns a real subprocess via `std::process::Command` and asserts the sandbox denies the forbidden operation
- Tests are fast (< 10s each); sandbox enforcement is exercised, not the full agent loop
- `#[cfg(target_os = "...")]` gates applied correctly; no-op on other OSes

---

### TEST-04: kay-tauri commands tested via tauri::test::mock_builder()

**Statement:** The three IPC command handlers (`start_session`, `stop_session`, `get_session_status`) are tested via Tauri's test harness (`tauri::test::mock_builder()`), not just by the memory canary.

**Acceptance criteria:**
- `crates/kay-tauri/tests/commands.rs` (or split per command) tests:
  - `start_session`: invokes, receives session_id, verifies AppState::sessions contains the session
  - `stop_session`: invokes start_session then stop_session, verifies token is cancelled
  - `get_session_status`: invokes for existing and non-existing session_id, verifies Running / Complete
- Uses `tauri::test::mock_builder().manage(AppState::default()).invoke_handler(...)`
- All three tests pass via `cargo test -p kay-tauri`

---

### TEST-05: kay-tauri UI has tauri-driver + WDIO smoke suite passing in CI

**Statement:** The `kay-tauri/ui/` frontend has a WebDriverIO test suite that launches the built Kay app, starts a session, and verifies the event stream renders.

**Acceptance criteria:**
- `@wdio/cli` + `@wdio/local-runner` installed in `crates/kay-tauri/ui/`
- `wdio.conf.ts` configured for `tauri-driver` (headless Chromium via ChromeDriver)
- Smoke test: app window appears, prompt input is visible, Run button is clickable, event stream renders at least one event
- Smoke test runs in CI on `macos-14` (primary); `ubuntu-latest` (secondary)
- **Open question (OQ-01):** macOS notarization requirements for `tauri-driver` in CI — see §11
- **Open question (OQ-02):** WebDriverIO version pinning strategy — see §11

---

### TEST-06: CI test matrix runs on ubuntu-latest + macos-14 + windows-latest

**Statement:** All `cargo test` jobs in the CI configuration run on all three OSes. Sandbox-specific tests gate on `runs-on` per the target OS.

**Acceptance criteria:**
- `ubuntu-latest` CI job: all tests (including `kay-sandbox-linux` subprocess tests)
- `macos-14` CI job: all tests (including `kay-sandbox-macos` subprocess tests)
- `windows-latest` CI job: all tests (including `kay-sandbox-windows` subprocess tests)
- Sandbox cross-platform jobs: conditionally enabled via `matrix` + `runs-on` gating
- All three CI jobs must be green for merge approval

---

### TEST-07: Coverage gate — CI fails if gap-list crate has zero tests post-merge

**Statement:** A CI check enumerates the gap-list crates and fails if any of them has no `tests/` directory or no `#[test]` functions.

**Acceptance criteria:**
- `scripts/coverage-gate.sh` script: iterates gap-list crates, checks for `tests/` dir + `*.rs` files under `tests/` containing `#[test]`
- Script exits 1 (fails CI) if any gap-list crate fails the check
- Script exits 0 if all gap-list crates pass
- Script runs on every PR and post-merge push
- Well-tested crates (kay-core, kay-tools, etc.) are NOT in the gap list and are excluded

## 10. Non-Negotiables

1. **DCO on every commit** — `Signed-off-by: Shafqat Ullah <shafqat@sourcevo.com>` on every commit. No exceptions.
2. **No direct main commits** — all changes via PR to `phase/09-multi-perspective-verification` or `phase/09.1-test-coverage` (branch name TBD per project conventions).
3. **TDD discipline** — RED commit (failing test) MUST precede GREEN commit (implementation) per wave. No skipping.
4. **No source changes to forge_\* crates** — tests are additive only. Do not refactor `forge_*` source code in this phase.
5. **RC version pins use `=` prefix** — any new RC dependencies added during this phase use `= "1.2.3-rc.4"` pinning.
6. **CI must be green on all 3 OSes** before merge. No partial green.
7. **Coverage gate is a hard CI failure** — post-merge push with a gap-list crate missing tests is a revert-worthy regression.

## 11. Open Questions

### OQ-01: tauri-driver macOS notarization in CI

`tauri-driver` spawns a GUI process on macOS. CI runners on `macos-14` may require notarization to avoid the "app is from an unidentified developer" dialog. Options:
- Use `INFO.plist` with `NSPrincipalClass = NSApplication` for the test binary
- Skip the WDIO macOS smoke suite on CI and run it locally only
- Use a headless CI configuration that does not require notarization

**Recommendation:** Start with `macos-14` (GitHub-hosted runners) which have GUI test support. If notarization fails in practice, fall back to local-only smoke testing on macOS.

### OQ-02: WebDriverIO version pinning strategy

WebDriverIO (`@wdio/cli`) releases frequently. Without a pin, CI can break on a minor version upgrade. Options:
- Pin to `^9.0.0` in `package.json` with an upper bound (`< 10.0.0`)
- Pin to exact `9.x.y` and update manually per phase
- Use `pnpm.overrides` in `package.json`

**Recommendation:** Pin to `^9.0.0` with a review step in the phase retro to update if a new major version is available. Do not pin to an exact patch version in `package.json` — minor updates are acceptable.

## 13. RC / Dependency Notes

No new RC dependencies are expected for this phase. All test tooling (`assert_cmd`, `proptest`, `trybuild`, `insta`, `tauri::test`, `tauri-driver`, `@wdio/cli`) uses stable versions already in the ecosystem.

If `tauri::test` APIs differ from the names used in Phase 9 (WARN-01 in the Phase 9 spec), the Wave 5 test implementation will be updated accordingly. The _concept_ (mock builder testing IPC commands) is stable; the _scaffolding_ is implementation-time detail.
