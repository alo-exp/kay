# Design: Phase 9.5 — TUI Frontend (ratatui)

**Date:** 2026-04-24
**Phase:** 9.5
**Requirements:** TUI-01, TUI-02, TUI-03, TUI-04, TUI-05
**Status:** Rev 1 — initial design

---

## 1. Problem Statement

Kay needs a full-screen terminal UI (TUI) consuming the same `kay-cli` JSONL event stream as the Tauri GUI (Phase 9). Users running over SSH, in low-bandwidth terminals, or by preference get a keyboard-driven, ANSI-256-color experience with live session view, tool-call inspector, and cost meter. No parallel agent-loop implementation — TUI is a pure frontend over the CLI contract.

---

## 2. Scope

### In scope (Phase 9.5)

- `crates/kay-tui/` ratatui-based TUI that spawns `kay run --jsonl` as a subprocess
- ratatui 0.27+ with `crossterm` backend (standard, non-WASM, non-Tauri backend)
- Multi-pane layout: session list + agent trace + tool-call inspector + cost meter
- Live event streaming from subprocess stdout via JSONL parser
- Keyboard-driven navigation (no mouse required; SSH-friendly)
- `kay tui` command dispatching from `kay-cli` (or standalone `kay-tui` binary)
- Session control via keyboard shortcuts (spawn / stop / fork)
- Dark theme via ANSI escape codes (16-color compatible with 256-color terminals)

### Out of scope (Phase 10)

Multi-session manager (Phase 10 adds spawn/pause/resume per-project), settings panel, API key management, OS keychain binding.

## 3. Architecture

### 3.1 Why Subprocess (not Embedded Agent Loop)

Phase 9's Tauri GUI uses `ipc::Channel<IpcAgentEvent>` for in-process IPC. The TUI uses a subprocess pipe for process-boundary IPC. Both frontends consume the same `kay-cli` JSONL event contract:

```
┌─────────────────────────────────────────────────────────────┐
│ kay run --jsonl --prompt "..." --persona forge             │
│     │                                                     │
│     ├──► JSONL stdout ──► kay-tauri (Tauri GUI)           │
│     │                    ipc::Channel<IpcAgentEvent>        │
│     │                                                     │
│     └──► JSONL stdout ──► kay-tui (ratatui TUI)            │
│                          subprocess pipe                    │
└─────────────────────────────────────────────────────────────┘
```

This is intentional: the CLI binary (`kay`) is the single source of truth for the event contract. Both GUI and TUI are consumers of that contract, not co-implementers of the agent loop.

### 3.2 TUI Event Types

The TUI consumes the same `IpcAgentEvent` JSONL events that the Tauri GUI receives via `Channel<T>`. The TUI deserializes them with `serde_json`:

```rust
// crates/kay-tui/src/events.rs
use serde::{Deserialize, Serialize};

// IpcAgentEvent must be re-defined here (kay-tui links to kay-core,
// not kay-tauri). This is intentional — TUI is a consumer of the CLI
// contract, not a re-implementation of IPC concerns.
//
// NOTE: Keep this type definition in sync with kay-tauri/src/ipc_event.rs.
// Any additive change to IpcAgentEvent should be mirrored here.
//
// serde_json tag/content naming mirrors kay-tauri's #[serde(tag="type", content="data")]
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", content = "data")]
pub enum TuiEvent {
    #[serde(rename = "TextDelta")]
    TextDelta { content: String },
    #[serde(rename = "ToolCallStart")]
    ToolCallStart { id: String, name: String },
    #[serde(rename = "ToolCallDelta")]
    ToolCallDelta { id: String, arguments_delta: String },
    #[serde(rename = "ToolCallComplete")]
    ToolCallComplete { id: String, name: String, arguments: serde_json::Value },
    #[serde(rename = "ToolCallMalformed")]
    ToolCallMalformed { id: String, raw: String, error: String },
    #[serde(rename = "Usage")]
    Usage { prompt_tokens: u64, completion_tokens: u64, cost_usd: f64 },
    #[serde(rename = "Retry")]
    Retry { attempt: u32, delay_ms: u64, reason: String },
    #[serde(rename = "Error")]
    Error { message: String },
    #[serde(rename = "ToolOutput")]
    ToolOutput { call_id: String, chunk: TuiToolOutputChunk },
    #[serde(rename = "TaskComplete")]
    TaskComplete { call_id: String, verified: bool, outcome: TuiVerificationOutcome },
    #[serde(rename = "ImageRead")]
    ImageRead { path: String, data_url: String },
    #[serde(rename = "SandboxViolation")]
    SandboxViolation { call_id: String, tool_name: String, resource: String, policy_rule: String, os_error: Option<i32> },
    #[serde(rename = "Paused")]
    Paused,
    #[serde(rename = "Aborted")]
    Aborted { reason: String },
    #[serde(rename = "ContextTruncated")]
    ContextTruncated { dropped_symbols: usize, budget_tokens: usize },
    #[serde(rename = "IndexProgress")]
    IndexProgress { indexed: usize, total: usize },
    #[serde(rename = "Verification")]
    Verification { critic_role: String, verdict: String, reason: String, cost_usd: f64 },
    #[serde(rename = "VerifierDisabled")]
    VerifierDisabled { reason: String, cost_usd: f64 },
    #[serde(rename = "Unknown")]
    Unknown { event_type: String },
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum TuiToolOutputChunk {
    #[serde(rename = "Stdout")]
    Stdout(String),
    #[serde(rename = "Stderr")]
    Stderr(String),
    #[serde(rename = "Closed")]
    Closed { exit_code: Option<i32>, marker_detected: bool },
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum TuiVerificationOutcome {
    #[serde(rename = "Pending")]
    Pending { reason: String },
    #[serde(rename = "Pass")]
    Pass { note: String },
    #[serde(rename = "Fail")]
    Fail { reason: String },
}
```

**Design note:** `TuiEvent` is a re-definition of `IpcAgentEvent` for the TUI consumer context. They are structurally identical (same JSONL wire format) but have different crate boundaries. Any change to the wire format must be reflected in both. This is analogous to how `kay-tauri` defines `IpcAgentEvent` as a mirror of `AgentEvent` — the mirroring is intentional for IPC isolation.

### 3.3 Subprocess Lifecycle

```
kay-tui startup
    │
    ├── parse CLI args (prompt, persona, session_id if resuming)
    │
    ├── spawn: tokio::process::Command::new("kay")
    │           .args(["run", "--jsonl", "--prompt", prompt, "--persona", persona])
    │           .stdout(Stdio::piped())
    │           .stderr(Stdio::inherit())  // kay logs go to terminal stderr
    │
    ├── read stdout line-by-line (AsyncReadExt::read_line)
    │       │
    │       ├── parse JSONL line → TuiEvent
    │       │
    │       └── dispatch to TuiApp state:
    │               EventLog::push(event)       // accumulate trace
    │               ToolMap::update(call_id)    // track active tool calls
    │               Metrics::accumulate(event)  // token/cost meter
    │
    ├── on user Ctrl+C or 'q': send SIGTERM to kay subprocess
    │       └── wait for exit, display summary
    │
    └── on kay exit: display final session summary
```

## 4. UI Design

### 4.1 Layout (ratatui Block layout)

```
┌────────────────────────────────────────────────────────────────────┐
│  KAY TUI                              [Ctrl+C] Stop  [q] Quit    │  ← Header (1 line)
├────────────────────────────────────────────────────────────────────│
│  │ Session: a1b2c3d4   Model: claude-4-opus   Persona: forge     │  │  ← Status Bar (1 line)
│  │ Tokens: 12,450↑ 3,200↓   Cost: $0.0042   Status: Running      │  │
│  ├────────────┬───────────────────────────────────────────────┤  │
│  │            │                                               │  │
│  │            │  ▶ hello, I'd like to help you with...        │  │  ← Agent Trace (main)
│  │            │                                               │  │
│  │            │  ┌─────────────────────────────────────────┐  │  │
│  │  Tool      │  │ 🔧 execute_commands                    │  │  │  ← Tool Call Card
│  │  Timeline  │  │   running...                           │  │  │
│  │            │  └─────────────────────────────────────────┘  │  │
│  │            │                                               │  │
│  │            │  [Usage: 12,450↑ 3,200↓ $0.0042]             │  │  │
│  │            │                                               │  │
│  ├────────────┴───────────────────────────────────────────────┤  │
│  │  > What would you like me to help with?                   │  │  ← Prompt Input (1 line)
│  │  [Enter] Send  [Ctrl+C] Stop  [Tab] Switch pane           │  │
│  └────────────────────────────────────────────────────────────┘
```

**Panes:**
1. **Header + Status Bar** (top 2 rows): session metadata + metrics + key bindings hint
2. **Tool Timeline** (left, narrow): horizontal bar of tool call bubbles, colored by name
3. **Agent Trace** (center, main): scrolling event log with auto-scroll on new events
4. **Prompt Input** (bottom): single-line input with history (up/down arrows)
5. **Optional Tool Inspector** (right panel, toggled): expands a tool-call's full arguments/output

### 4.2 Color Palette (ANSI 256-color)

| Element           | Color         | ANSI Code | Notes                    |
|-------------------|---------------|-----------|--------------------------|
| Background        | #1e1e2e       | `235`     | Dark charcoal             |
| Surface           | #2a2a3e       | `236`     | Slightly lighter          |
| Border            | #45475a       | `239`     | Subtle separator          |
| Text Primary      | #cdd6f4       | `255`     | Light gray                |
| Text Secondary    | #a6adc8       | `251`     | Muted                     |
| Text Muted        | #6c7086       | `243`     | Very muted                |
| Accent (tool)     | #89b4fa       | `75`      | Blue                      |
| Success           | #a6e3a1       | `78`      | Green                     |
| Error             | #f38ba8       | `204`     | Red                       |
| Warning/Retry     | #fab387       | `223`     | Orange                    |
| Tool Call         | #cba6f7       | `183`     | Purple                    |
| Usage/Cost        | #f9e2af       | `229`     | Yellow                    |

**Terminal compatibility:** All colors are standard 256-color ANSI. The layout degrades gracefully on 16-color terminals (falls back to bold/bright for emphasis, inverse for selection).

### 4.3 Key Bindings

| Key(s)              | Action                              | Pane           |
|---------------------|-------------------------------------|----------------|
| `↑` / `k`           | Scroll up in Agent Trace            | Trace          |
| `↓` / `j`           | Scroll down / resume auto-scroll    | Trace          |
| `Tab`               | Cycle focus: Trace → Timeline → Inspector | Global |
| `Enter`             | Focus prompt / send prompt          | Prompt         |
| `Ctrl+C`            | Stop current session (SIGTERM kay) | Global         |
| `q`                 | Quit TUI                           | Global         |
| `Ctrl+L`            | Clear trace                         | Trace          |
| `t`                 | Toggle Tool Timeline visibility     | Global         |
| `↑` / `↓` in prompt | Command history (up/down)          | Prompt         |
| `Esc`               | Cancel current input / close popup  | Global         |

### 4.4 Keyboard Navigation Rules

- **Mouse not required** — all actions have keyboard equivalents
- **Focus is explicit** — Tab cycles through panes; active pane has bright border
- **Auto-scroll** — Agent Trace auto-scrolls to bottom on new events unless user scrolls up
- **SSH-safe** — no mouse events; no Unicode beyond basic multilingual plane (emoji OK for status indicators)

### 6.2 kay-cli Extension

The `kay tui` subcommand spawns `kay-tui` binary. If `kay-tui` is invoked standalone, it spawns `kay run --jsonl` as a subprocess:

```rust
// In kay-cli src/commands.rs (extend existing run command)
#[derive(Parser, Debug)]
pub enum TuiCommand {
    /// Launch the TUI frontend.
    ///
    /// Spawns `kay run --jsonl` as a subprocess and renders events
    /// in a full-screen ratatui terminal UI.
    Tui {
        /// Prompt to run immediately on startup.
        #[arg(short, long)]
        prompt: Option<String>,

        /// Persona to use (forge, sage, muse).
        #[arg(short, long, default_value = "forge")]
        persona: String,
    },
}
```

## 7. Non-Negotiable Constraints

1. **Keyboard-only navigation** — no mouse required; SSH-friendly
2. **Subprocess consumption** — `kay-tui` consumes `kay run --jsonl` JSONL, NOT in-process `AgentEvent` channel
3. **Same event contract** — `TuiEvent` must be structurally identical to `IpcAgentEvent` (serde rename attrs ensure wire compatibility). NOTE: Any additive change to the event wire format must be mirrored in both `kay-tauri/src/ipc_event.rs` and `kay-tui/src/events.rs`. Document this in both files.
4. **Ratatui 0.27+** — use stable ratatui; verify 0.27.x API at execution time
5. **DCO on every commit** — `Signed-off-by: Shafqat Ullah <shafqat@sourcevo.com>`
6. **Branch: `phase/09.5-tui-frontend`** — PR only, no direct main commits
7. **100% TDD** — RED commit before GREEN commit per wave
8. **No key logging** — TUI subprocess inherits terminal; if kay run logs to stdout, TUI filters it out (JSONL only)
9. **EventLog cap** — `EventLog::push` evicts oldest event when cap (10,000) is reached
10. **JSONL malformed lines** — `JsonlIterator` skips non-JSON lines with `tracing::debug!` (ERR-01 fix)
